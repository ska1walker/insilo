Lernpfad · Olares verstehen

Wie funktioniert eigentlich Insilo?

Insilo lebt in einer kleinen Box im Serverraum. Diese Box heißt Olares. Auf den nächsten Seiten erklären wir Ihnen, wie diese Box im Inneren aufgebaut ist — Schritt für Schritt, mit Bildern und ohne unnötige Fachwörter. Wenn Sie eine Aufnahme aus der App heraus starten, wandert sie durch viele kleine Zimmer und über viele kurze Wege. Wir zeigen Ihnen, wo sie überall vorbeimuss.

Lesedauer ≈ 12 Minuten Keine Vorkenntnisse nötig Schreibstil: Sie-Form, ruhig, sachlich

Modell · Die Bürohaus-Analogie

Olares ist ein digitales Bürogebäude.

Stellen Sie sich Olares vor wie ein Bürohaus, das bei Ihnen im Serverraum steht. Im Inneren wohnen verschiedene Mieter — jeder Mieter ist eine App. Insilo ist einer dieser Mieter, andere wären beispielsweise die Sprachmodell-Gateway LiteLLM oder ein Notiz-Tool.

Jeder Mieter hat sein eigenes Stockwerk. Im Stockwerk gibt es Räume. In jedem Raum stehen Geräte. Und es gibt eine Telefonzentrale, die alle Räume miteinander verbindet — sowohl innerhalb desselben Stockwerks als auch zwischen den Stockwerken.

Stockwerk → Namespace
Raum → Pod
Gerät im Raum → Container
Telefonzentrale → Service · Cluster-DNS
Schlüsselkarten an Türen → NetworkPolicy
Lagerraum neben dem Stockwerk → Volume · hostPath

Diese sechs Bilder begleiten Sie durch das ganze Dokument. Wenn ein Begriff Sie kalt erwischt — die Tooltips beim Hovern sind Ihre Eselsbrücke.

Konzept 01 · Namespaces

Stockwerke: eine pro App.

Wenn Sie eine App auf Olares installieren, legt Olares automatisch ein neues Stockwerk an — einen Namespace. Insilos Stockwerk heißt zum Beispiel insilo-kaivostudio. Der zweite Teil kaivostudio ist Ihr Olares-Username — so weiß die Box, wem das Stockwerk gehört.

Warum getrennte Stockwerke?

  • Damit eine App nicht versehentlich auf die Sachen einer anderen zugreift.
  • Damit Sie einen Mieter rauswerfen können (App uninstall), ohne dass die anderen davon etwas merken.
  • Damit Olares pro Stockwerk eigene Sicherheitsregeln vergeben kann.

So sieht das auf einer typischen Box aus

  • insilo-kaivostudio — unsere App
  • litellm-kaivostudio — die Sprachmodell-Gateway
  • user-system-kaivostudio — die geteilten System-Datenbanken (Postgres, Redis)
  • user-space-kaivostudio — der Olares-Hauptbenutzer und seine Files
  • os-platform — Olares' eigene Plattform-Dienste
Im Terminal nachschauen 
$ kubectl get ns
NAME                       STATUS   AGE
insilo-kaivostudio         Active   2d
litellm-kaivostudio        Active   3d
user-system-kaivostudio    Active   4d
os-platform                Active   4d

Jede Zeile ist ein Stockwerk. Olares hat dazu noch zusätzliche Stockwerke für interne Aufgaben — die meisten davon brauchen Sie nicht zu kennen.

Konzept 02 · Pods & Container

Räume und Geräte: was wirklich läuft.

Ein Pod ist ein Raum im Stockwerk. In diesem Raum steht meistens genau ein Gerät — der Container. Manchmal stehen aber auch zwei oder drei Geräte zusammen in einem Raum, wenn sie eng zusammenarbeiten sollen. Das nennt man dann einen Multi-Container-Pod.

Die fünf Räume von Insilo

  • Frontend-Raum. Hier steht die Web-Oberfläche (PWA). Zusätzlich steht ein Türsteher in diesem Raum — der Envoy-Sidecar, der prüft, ob ein Besucher reinkommen darf.
  • Backend-Raum. Hier rechnet der API-Server (FastAPI). Er hat keinen Envoy davor — interne Anfragen vom Frontend sollen direkt durch.
  • Worker-Raum. Hier laufen Hintergrund-Jobs (Celery): Transkription, Zusammenfassung, Embedding.
  • Whisper-Raum. Hier transkribiert OpenAIs Sprachmodell large-v3 die Aufnahmen.
  • Embeddings-Raum. Hier wandelt das BGE-M3-Modell Text in Vektoren um — die Grundlage für die „Fragen"-Funktion.

Wenn ein Container abstürzt, baut Olares automatisch einen neuen — der Raum bleibt, das Gerät wird ersetzt. Wenn das ganze Stockwerk neu gebaut wird (also die App neu installiert), kann der Pod an einer anderen physischen Stelle landen — aber das merken Sie nie, weil die Telefonzentrale den Pod trotzdem findet. Das ist Kapitel 3.

Konzept 03 · Services & Cluster-DNS

Die Telefonzentrale: wie zwei Pods sich finden.

Damit zwei Pods miteinander reden können, brauchen sie eine Adresse. Pods bewegen sich aber von Zeit zu Zeit (Restarts, Updates, Knotenausfälle). Deshalb gibt Olares jedem Pod eine Service-Adresse, die bleibt — wie eine interne Telefonnummer, die einen Mitarbeiter erreicht, egal an welchem Schreibtisch er gerade sitzt.

Die volle Form einer Service-Adresse lautet:

<service-name>.<namespace>.svc.cluster.local

Konkret bei uns: das Backend ist im Stockwerk insilo-kaivostudio als Service insilo-backend registriert. Seine volle Adresse ist also: 

insilo-backend.insilo-kaivostudio.svc.cluster.local

Innerhalb desselben Stockwerks reicht der Kurzname: http://insilo-backend:8000. Das ist es, was Insilos Frontend benutzt, wenn es eine Browser-Anfrage an das Backend weiterleitet.

Im Terminal nachschauen 
$ kubectl get svc -n insilo-kaivostudio
NAME                TYPE        CLUSTER-IP       PORT(S)    AGE
insilo              ClusterIP   10.233.36.12     3000/TCP   2d
insilo-backend      ClusterIP   10.233.36.45     8000/TCP   2d
insilo-whisper      ClusterIP   10.233.36.51     8001/TCP   2d
insilo-embeddings   ClusterIP   10.233.36.79     8002/TCP   2d

Konzept 04 · NetworkPolicies

Schlüsselkarten: wer darf wo hin.

Nur weil zwei Pods sich theoretisch finden könnten, dürfen sie noch lange nicht miteinander reden. Das regeln die NetworkPolicies — eine Art Schlüsselkarten an den Türen jedes Stockwerks. Olares legt davon ziemlich strenge an: standardmäßig darf nichts rein, und es muss explizit erlaubt werden.

Konkretes Beispiel: das System-Datenbank-Stockwerk user-system-kaivostudio lässt nur Stockwerke rein, die das Label bytetrade.io/ns-owner=kaivostudio tragen. Olares setzt dieses Label automatisch, sobald eine App Ihnen gehört.

Die Henne-Ei-Falle, die uns zwei Tage gekostet hat

Olares setzt das ns-owner-Label erst nachdem der Install komplett durchgelaufen ist. Aber unser Backend möchte beim ersten Start direkt mit der Datenbank reden — ohne Label kein Zugriff. Die ersten Versuche, die DB-Migration als Helm-Hook laufen zu lassen, sind genau daran gescheitert.

Lösung: die Migration läuft jetzt als Init-Container im Backend-Pod selbst, mit einem Retry von 60×5 Sekunden. Pod-Restarts überbrücken einfach den Label-Window — wenn das Label da ist, kommt der Connect durch und die Migrationen werden eingespielt.

Konzept 05 · Volumes & hostPath

Lagerräume: damit Daten überleben.

Ein Container ist von Natur aus vergesslich. Wenn er stirbt und neu startet, sind alle Dateien weg, die er zwischendurch geschrieben hat — als hätten Sie einen Laptop in den Werkszustand zurückgesetzt. Damit Daten überleben, gibt es Volumes — Lagerräume außerhalb des Containers.

Drei Typen, die für uns wichtig sind

  • hostPath. Ein Ordner auf der Olares-Box selbst. Olares legt für jede App einen festen Pfad an, z. B. /olares/userspaces/kaivostudio/Data/insilo. Egal wie oft der Pod stirbt, der Ordner bleibt.
  • ConfigMap. Eine kleine Textdatei, die Olares aus unseren Helm-Templates erzeugt und in den Pod hängt. Wir benutzen das z. B. für die SQL-Migrations-Dateien.
  • PersistentVolumeClaim. Ein vom Cluster zugeteilter Speicherblock. Olares macht das für seine System-Dienste automatisch; wir nutzen ihn aktuell nicht direkt.

Wo Insilos Audio konkret landet

Audio-Dateien landen in einem hostPath. Die App schreibt intern auf /app/data/audio/<org_id>/<meeting_id>.webm. Olares mappet das transparent auf /olares/userspaces/kaivostudio/Data/insilo/audio/… auf der physischen Box. Jedes Pod-Restart, jedes Upgrade — das Audio bleibt liegen. Es verlässt diese Festplatte niemals.

Lifecycle · Schritt-für-Schritt

Was passiert, wenn Sie auf „Aufnahme starten" drücken?

Eine Aufnahme reist durch sechs Stationen. Klicken Sie auf eine Stufe, um zu sehen, was an dieser Station geschieht.

Browser nimmt auf

Sie klicken in der PWA auf Start. Der Browser fragt das Mikrofon an (navigator.mediaDevices.getUserMedia), öffnet einen MediaRecorder und sammelt Audio-Chunks im Speicher. Die goldene Pulse-Linie am oberen Bildschirmrand zeigt, dass die Aufnahme läuft. Bis zu diesem Punkt ist noch nichts auf der Box.

Mikrofon → MediaRecorder → Audio-Blob im Browser

Upload zum Backend

Sie klicken auf Stopp. Der Browser baut ein FormData-Paket aus dem Audio-Blob und schickt es an POST /api/v1/recordings. Die Anfrage geht zuerst an den Frontend-Pod (Same-Origin), trifft dort den Envoy- Sidecar, der Ihr Authelia-Cookie prüft, und wird dann von Next.js' rewrites() intern an den Backend-Pod weitergeleitet — über die Service-Adresse http://insilo-backend:8000. Ihr Browser sieht davon nichts; aus seiner Sicht ist alles eine HTTP-Anfrage an die gleiche Domain.

Browser → Frontend + Envoy → Backend-Pod

Backend speichert

Das Backend liest den X-Bfl-User-Header (von Envoy injiziert), erzeugt eine UUID für das Meeting, schreibt das Audio in den hostPath unter /app/data/audio/<org>/<id>.webm und legt einen Eintrag in der Postgres-Tabelle public.meetings mit Status queued an. Anschließend stellt es eine Celery-Aufgabe in die Redis-Queue und antwortet dem Browser sofort mit 201 Created.

hostPath /app/data Postgres meetings Redis-Queue

Worker transkribiert

Der Worker-Pod horcht permanent an der Redis-Queue. Er nimmt die Aufgabe an, liest das Audio aus dem hostPath, POSTet es an den Whisper-Service (http://insilo-whisper:8001/transcribe) und wartet auf das Transkript zurück. Der Status des Meetings wird unterwegs von queued auf transcribing auf transcribed hochgezogen. Das Frontend pollt alle 2 Sekunden, um den Fortschritt anzuzeigen.

Worker → Whisper-Pod → Postgres transcripts

LLM fasst zusammen

Sobald das Transkript steht, lädt der Worker die konfigurierte Vorlage (System-Prompt + JSON-Schema), fügt das Transkript an und schickt eine OpenAI-kompatible Chat-Completion-Anfrage an den konfigurierten LLM-Endpunkt. Standardmäßig ist das die Olares-LiteLLM-Gateway im Nachbar-Stockwerk; Sie können in den Einstellungen aber jeden anderen Endpunkt eintragen (eigenes Ollama, OpenAI-API, …). Die strukturierte JSON-Antwort landet in public.summaries.

Worker → LiteLLM (oder externer LLM) → Postgres summaries

Embeddings + fertig

Als letzte Stufe zerlegt der Worker das Transkript in kurze Abschnitte (Chunks), schickt jeden an den BGE-M3-Embedding- Service und speichert die Vektoren in public.meeting_chunks. Damit ist das Meeting für die „Fragen"-Funktion durchsuchbar. Der Status springt auf ready, das Frontend hört auf zu pollen, und Sie sehen Transkript und Zusammenfassung.

Worker → Embeddings-Pod → pgvector

Erfahrung · Was wir bezahlt haben

Sechs Stolpersteine, die uns Tage gekostet haben.

Diese Liste ist die Quintessenz aus den Versionen v0.1.7 bis v0.1.21. Wenn Sie eine eigene App auf Olares bringen, beachten Sie diese sechs Punkte — sonst zahlen Sie das gleiche Lehrgeld.

  1. Henne und Ei beim Install

    SymptomHelm-Post-Install-Hook scheitert mit Timeout — Pod kommt nie an die Datenbank.

    LösungMigrationen als Init-Container der App selbst laufen lassen, nicht als Helm-Hook. Olares setzt das benötigte ns-owner-Label erst nach erfolgreichem Install.

  2. .Files.Get ist nicht erlaubt

    SymptomChart-Upload an Olares' Market-Renderer schlägt fehl: „can't evaluate field Files".

    LösungSQL inline in den ConfigMap-Template baken. Generator-Script scripts/regen-migrations.py automatisiert das, Drift-Check im CI verhindert vergessene Updates.

  3. runAsInternal: true ist Studio-Only

    SymptomFrontend Init:CrashLoopBackOff („services aren't ready in 20s") nach harmlos klingender Manifest-Änderung.

    LösungFinger weg von runAsInternal. Per-Org Settings (User wählt LLM-Endpoint selbst) lösen das ursprüngliche Cross-NS-Problem viel sauberer.

  4. psql -f hängt

    SymptomMigration-Job läuft, druckt „Applying file.sql..." und hängt unendlich, ohne Fehler.

    LösungPython + asyncpg statt psql. Jeder File als ein conn.execute(sql) — eine implicit transaction, keine Splitting-Logik.

  5. MinIO ist nicht erreichbar

    SymptomAudio-Upload kracht mit HTTP 500, S3-Client kommt nicht zur Olares-eigenen MinIO durch.

    LösungLocal-Filesystem-Storage-Backend, Audio in hostPath. Audio-Dateien bleiben auf der Box — passt zum Datensouveränitäts-Pitch sogar besser als jeder S3.

  6. Vier Version-Stellen müssen synchron sein

    SymptomOlares' Market-Manager beschwert sich beim Upload mit unklarer Version-Mismatch-Fehlermeldung.

    LösungMarc's Golden Rule: Chart.yaml.version = appVersion = OlaresManifest.metadata.version = spec.versionName. Bei uns automatisiert via scripts/release.sh + Pre-flight-Check in CI.

Nachschlagen · Glossar

Begriffe, die in diesem Dokument vorkommen.

App
Mieter im Olares-Bürogebäude. Bringt eigene Pods, Services und Templates mit.
Authelia
Der Türsteher von Olares. Validiert Login-Cookies, bevor jemand zur App durchgelassen wird.
Backend
Der Server, der die App-Logik hat. Bei Insilo: FastAPI.
Celery
Python-Bibliothek für Hintergrund-Jobs. Holt sich Aufgaben aus einer Warteschlange (Redis bei uns) und arbeitet sie ab.
Citus
Die Postgres-Variante, die Olares als System-DB benutzt. Unterstützt verteilte Tabellen.
Cluster-DNS
Die interne Telefonzentrale. Übersetzt Service-Namen wie insilo-backend in tatsächliche Pod-IPs.
ConfigMap
Eine kleine Konfigurationsdatei, die Olares aus Helm-Templates erzeugt und in einen Pod hängt.
Container
Ein verpacktes Programm, das in einem Pod läuft. Bringt seine eigene Bibliothek mit.
DSGVO
EU-Datenschutz-Grundverordnung. Unser Hauptgrund, dass nichts in die Cloud darf.
Envoy
Der Türsteher-Container, den Olares automatisch vor manche Pods stellt. Prüft das Authelia-Cookie.
GHCR
GitHub Container Registry. Dort liegen die Docker-Images von Insilo, öffentlich zugänglich.
Helm
Der Paketmanager für Kubernetes. Beschreibt eine App als Sammlung von Templates mit Variablen.
hostPath
Ein Ordner auf der Olares-Maschine selbst, in den Pods schreiben dürfen. Überlebt Pod-Restarts.
k3s
Die schlanke Kubernetes-Variante, auf der Olares aufbaut. Ein Single-Binary, ~50 MB.
Kubernetes
Das Betriebssystem für Container-Cluster. Plant, startet und überwacht Pods.
LiteLLM
Ein Gateway, das viele Sprachmodelle hinter einer OpenAI-kompatiblen API zugänglich macht.
Manifest
Die OlaresManifest.yaml — beschreibt einen App-Eintrag im Olares-Markt.
Namespace
Ein Stockwerk im Olares-Gebäude. Pods, Services und Policies sind jeweils einem Namespace zugeordnet.
NetworkPolicy
Die Schlüsselkarten-Regel. Sagt, welche Pods in welchen Namespaces eingehende Verbindungen erlauben.
Olares
Das selbst-hostbare Cluster-Betriebssystem, auf dem Insilo läuft.
pgvector
Postgres-Erweiterung für Vektor-Suche. Brauchen wir für die „Fragen"-Funktion.
Pod
Ein Raum im Stockwerk. Enthält einen oder mehrere Container, die sich Netzwerk und Volumes teilen.
PWA
Progressive Web App. Eine Website, die sich wie eine App anfühlt — installierbar, offline-fähig.
Service
Die feste Adresse, unter der ein Pod im Cluster erreichbar ist. Bleibt stabil, auch wenn Pods restarten.
Sidecar
Ein zusätzlicher Container im selben Pod (z. B. der Envoy-Sidecar vor dem Frontend).
Whisper
Das Open-Source-Spracherkennungsmodell von OpenAI. Läuft bei uns lokal auf der Box.
